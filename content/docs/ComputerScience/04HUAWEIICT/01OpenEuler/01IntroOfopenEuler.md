## Unix发展历程




### 早期历史与贝尔实验室
> Linux操作系统源自著名的Unix操作系统。Unix操作系统的历史，可以追溯到1965年由贝尔实验室和MIT以及通用电气GE合作开发的**Multics(Multiplexed Information and Computing Service，多路信息和计算服务)操作系统**。1969年，贝尔实验室因为对该系统不满而退出合作。肯·汤普森(Ken Thompson)和丹尼斯·里奇(Dennis Ritchie)是贝尔实验室参与该项目的两位工程师。（后文用K,D代称）



最初的Unix操作系统是用B语言和汇编语言开发的。B语言是以**基本组合编程语言(Basic Combined Programming Language, BCPL)** 为蓝本(取名自其首字母)，由K&D二人在1969年左右开发的一种编译型语言，目的是在PDP-7小型计算机上编写操作系统和其他系统软件。而Unix系统正是K&D基于PDP-7计算机开发的新系统，作为复杂的Multics操作系统的简化替代品，并在1970年命名为Unix操作系统。'Unix'是对'Multics'的双关语，意为采用简单的方法实现了分时多任务。

为了便于将Unix移植到其他的不同于PDP-7的计算机上，1972年丹尼斯里奇创造了C语言(取名自BCPL的第二个字母)，继承了B语言的许多思想，加入了数据类型的概念以及其他特性，是一种功能更加强大、更加通用的编程语言。

1973年Unix被用C语言进行重写，这一举措提高了其可移植性，为后续被广泛采用奠定基础。


基于美国政府与美国电话电报公司（AT&T）的反垄断协议，AT&T被禁止进入非电信业务领域，特别是电子计算机市场。贝尔实验室无法将Unix商业化。

Unix操作系统的首篇论文 **"The UNIX Time-Sharing System"** by **Ken Thompson** and **Dennis M. Ritchie** (1974)，介绍了Unix的设计理念、结构和特点，并描述了其在PDP-11上的实现。论文详细讨论了文件系统、进程管理和开发环境的设计等相关内容。

1978年出版的书名为 **"The C Programming Language"** by **Brian W. Kernighan** and **Dennis M. Ritchie** 的书，不仅详述了C语言的程序设计，还间接描述了Unix的许多技术细节。

### 学术界的传播与BSD的诞生

20世纪70年代，美国高校的计算机专业缺乏《操作系统》课程的教学实践案例。在Unix的首篇论文"The UNIX Time-Sharing System"发表之后，Unix被教育界所熟知，纷纷向贝尔实验室询问Unix操作系统的状况。贝尔实验室最终决定向高校开放Unix操作系统，作为《操作系统》课程的教学实例。因此到1975年，Unix的第六版在学术界和研究环境中广受欢迎。1977年，加州大学伯克利分校发布了改进后的Unix版本——**伯克利软件分发（Berkeley Software Distribution, BSD）**。BSD引入了许多创新，如`csh` shell、`vi`编辑器、TCP/IP网络协议栈。BSD的发展对Unix操作系统的扩展和改进有着重要的影响，并成为现代Unix和Unix-like系统的基础，包括**FreeBSD**、**NetBSD**和**OpenBSD**。

### 商业化与“Unix战争”

20世纪80年代，Unix开始商业化使用。**AT&T的System V**和**加州大学伯克利分校的BSD**成为Unix开发的两大主要分支。这些分支经常竞争，导致了不同商业供应商之间的所谓“Unix战争”。

**Unix国际 (Unix International)** 是由AT&T和Sun Microsystems等公司于1988年成立的联盟，旨在推广**System V Release 4 (SVR4)** 作为标准的Unix版本。SVR4结合了AT&T的System V和Sun Microsystems的BSD系统的特点，试图统一Unix市场。Unix国际的目标是通过标准化Unix操作系统来减少分裂和不兼容问题，从而促进Unix在商业领域的普及。

为了对抗Unix国际和SVR4的影响，特别是由于担心AT&T和Sun Microsystems的主导地位可能会削弱其他厂商的竞争力，一些计算机公司，包括**IBM**、**HP**、**DEC**等，组建了**开放软件基金会 (Open Software Foundation, OSF)**。OSF开发了自己的Unix版本，称为OSF/1，希望通过提供一个开放的、标准化的Unix版本来吸引更多的用户和开发者。

#### “Unix战争”的主要竞争方面

这两大阵营的竞争被称为“Unix战争”，主要集中在以下几个方面：

- **标准化**：Unix国际和OSF都试图通过各自的版本来设定Unix的标准，吸引更多的厂商和用户。
- **市场份额**：每个阵营都希望通过提供更优的技术和支持来扩大其在Unix市场的份额。
- **技术创新**：这种竞争推动了Unix技术的不断创新和改进。例如，SVR4结合了System V和BSD的优点，而OSF/1则引入了许多新的功能和改进。

### 战争的缓和与新的挑战

尽管竞争激烈，这场“Unix战争”最终推动了Unix技术的进步，并且在一定程度上促成了Unix操作系统的标准化。到1993年，随着**Novell**收购**Unix System Laboratories (USL)**，并将Unix的商标和规范转让给开放组织 **The Open Group**，竞争逐渐缓和下来。The Open Group后来发布了单一Unix规范 **(Single UNIX Specification)**，进一步推动了Unix系统的兼容性和标准化。

20世纪90年代初期，Unix操作系统面临来自微软的**Windows NT**操作系统的激烈竞争，也导致了两大Unix阵营之间的“Unix战争”逐渐平息下来。

近20年以来，随着**Linux**操作系统的兴起，商业Unix（**AIX**、**HP-UX**、**Solaris**、**IRIX**、**SCO Unix**等）逐渐退出了主流市场。

## Linux发展历程

### Linux历史

Linux 操作系统是 Linux 内核和 GNU 项目（GNU's Not Unix）的结合物。

Linux 操作系统深受 Minix 操作系统​的影响。Minix 操作系统是由安德鲁·塔能鲍姆（Andrew S. Tanenbaum）教授在1987年开发的一个基于 Unix 的教学操作系统。林纳斯·托瓦兹（Linus Torvalds）在创建 Linux 时，受到了 Minix 的启发并借鉴了其设计理念。以下是一些关键点：

*   **启发与动机**：Linus Torvalds 在学习 Minix 的过程中，受到其开源理念的启发，并萌生了创建一个类似但功能更强大、自由可用的操作系统的想法。
*   **设计理念**：Minix 是一个微内核操作系统，这意味着其核心功能最小化，只处理最基本的任务，而其他功能则通过外部模块实现。虽然 Linux 最终选择了单内核（Monolithic kernel）结构，但其早期的设计理念和代码的组织受到了 Minix 的影响。
*   **学习与实践**：通过研究 Minix 的源码，Linus Torvalds 学习了操作系统的设计与实现，这为他开发 Linux 提供了宝贵的实践经验。
*   **开源社区**：Minix 的开源特性促进了用户和开发者的广泛参与，这也为后来的 Linux 社区建设提供了重要的经验参考。

Linus Torvalds 在使用 Minix 操作系统的过程中，发现它在某些方面存在局限性和不足，尤其是在性能和功能方面，不适合作为一个全面的操作系统使用，更适合用于操作系统教学。此外由于 Minix 操作系统有更为严格的授权限制，导致 Linus Torvalds 决定自己动手编写一个新的操作系统内核，以满足自己的需求和兴趣。下面是 Linux 内核初期的发展情况：

*   **1991 年**: Linus Torvalds 开始编写新的操作系统内核。
*   **1991 年 8 月 25 日**: Linus Torvalds 在 comp.os.minix 新闻组上发布了一则消息，宣布他正在开发一个新的免费操作系统，并希望得到反馈和建议。这封著名的邮件以“只是一个爱好，不会大到像 GNU 那样专业和庞大”结束。
*   **1991 年 9 月 17 日**: Linus 发布了 Linux 内核的第一个版本，版本号为 0.01。这个版本没有任何图形界面，功能非常有限，主要包括基本的​​文件系统操作和进程管理。
*   **1991 年 10 月 5 日**: Linus 发布了 Linux 内核的 0.02 版本，这是第一个可以运行 bash shell 和 gcc 编译器的版本，标志着 Linux 开始具备基本的操作系统功能。在这次发布中，Linus 邀请其他程序员参与开发和测试，这标志着 Linux 开始向开源社区开放。随着越来越多的开发者参与进来，Linux 内核得以迅速发展和完善。
*   **1992 年**: Linus 发布了 Linux 内核版本 0.12，首次使用了 GNU 通用公共许可证（GPL），确保了其开源和自由软件的性质。
*   **1994 年 3 月 14 日**: Linux 内核的第一个正式稳定版本 1.0 发布。这标志着 Linux 已经从一个实验性的项目发展成为一个功能全面的操作系统内核。

### GNU 项目

GNU 项目起源于1983年理查德·斯托曼（Richard Stallman）发布的 GNU 宣言，旨在创建一个完全自由的类 Unix 的操作系统，包括所有必要的工具和组件，使用户能够在不受限制的情况下使用计算机。

GNU 项目包含如下主要组件：

*   **GNU C 编译器 (GCC)**：是一个重要的编译系统，支持多种编程语言，是 GNU 项目的关键组件之一。
*   **GNU Bash Shell**：一个流行的命令行解释器，广泛用于 Unix 类系统。
*   **核心库 (glibc)**：GNU C 库是所有 GNU 程序所依赖的标准库，为系统调用和基本功能提供支持。
*   **GNU Make**：构建自动化工具。
*   **GNU Emacs**：文本编辑器。
*   **GNU Core Utilities**：基本命令行工具。

GNU 软件通常使用 GNU 通用公共许可证（GPL）。GPL 确保了软件的自由性，规定任何人都可以自由使用、修改和分发 GPL 软件，但必须将修改后的软件也以 GPL 许可证发布，从而保证了自由软件的持续发展。

作为 GNU 操作系统的一部分，GNU 项目还开发了 Hurd 内核，旨在替代 Unix 内核。然而，Hurd 内核项目进展缓慢，导致大多数 GNU 工具与 Linux 内核结合使用，形成了我们今天所熟知的 GNU/Linux 操作系统。

### 主流 Linux 发行版

目前市场上主要有以下广泛使用的 Linux 发行版：

*   **Debian**：由全球社区开发的一个自由的发行版，以其稳定性和安全性著称。其特点是具有稳定的发布周期，注重系统的稳定性和安全更新，适合开发者、系统管理员和高级用户。发行版包括 Debian Stable（稳定版）、Debian Testing（测试版）和 Debian Unstable（不稳定版）。
*   **Fedora**：由红帽公司赞助的一个前沿发行版，旨在提供最新的开源技术。其特点是具有快速更新的发布周期，注重创新和新技术的集成，适合开发者和技术爱好者。发行版包括 Fedora Workstation（桌面版）和 Fedora Server（服务器版）。
*   **商业Linux发行版Red Hat Enterprise Linux (RHEL)**：由Red Hat公司开发的一个商业发行版，旨在为企业级环境提供高度稳定和安全的操作系统。具有严格的发布周期和长期支持版本，注重企业安全、可靠性和可扩展性，适合企业用户和专业IT管理员。
*   **商业Linux发行版SUSE Linux Enterprise Server (SLES)**：提供类似RHEL的企业级支持和功能，特别强调在欧洲市场的适用性。
*   **Ubuntu**：一个基于 Debian、用户友好的发行版，旨在为桌面和服务器提供一个易于使用的操作系统。具有活跃的社区支持，定期发布长期支持版本（Long Term Support, LTS），适合初学者和普通用户。发行版包括 Ubuntu Server（服务器版）和 Ubuntu Desktop（桌面版）。
*   **CentOS (Community ENTerprise Operating System)**：原本是 Red Hat Enterprise Linux（RHEL）的社区版，提供与 RHEL 几乎完全兼容的免费平台。它以稳定性和企业级功能为主要特点，适合需要企业级操作系统但无需付费支持的用户和组织。自2021年起，CentOS 转变为 CentOS Stream，成为 RHEL 的直接上游，提前提供和测试即将在 RHEL 中推出的软件包。

## openEuler发展历程


openEuler是一个开源的企业级Linux操作系统，由华为发起并支持，目前由**开放原子开源基金会**（OpenAtom Foundation）孵化及运营，旨在提供一个稳定和安全的操作系统平台，支持多种硬件架构。openEuler 强调其在云计算和边缘计算领域的应用，提供了丰富的软件包和强大的性能优化。openEuler以其高性能、安全性和易用性为主要特点，适合需要高效企业级操作系统的用户和组织。

openEuler操作系统经历的里程碑阶段如下：

*   **2019年12月31日**，面向多样性计算的操作系统开源社区openEuler正式成立。
*   **2020年3月30日**，openEuler 20.03 LTS版本正式发布，为Linux世界带来一个全新的具备独立技术演进能力的Linux发行版。
*   **2020年9月30日**，首个openEuler 20.09创新版发布，该版本是openEuler社区中的多个公司、团队、独立开发者协同开发的成果，在openEuler社区的发展进程中具有里程碑式的意义，也是中国开源历史上的标志性事件。
*   **2021年3月31日**，发布openEuler 21.03内核创新版，该版本将内核升级到5.10，还在内核方向实现内核热升级、内存分级扩展等多个创新特性，加速提升多核性能，构筑千万核运管能力。
*   **2021年9月30日**，全新openEuler 21.09创新版如期而至，这是欧拉首个支持全场景的社区版。增强服务器和云计算的特性，发布面向云原生的业务混部CPU调度算法、容器化操作系统KubeOS等关键技术；同时发布边缘和嵌入式版本。
*   **2021年11月9日**，华为公司将openEuler捐赠给开放原子开源基金会。这一捐赠标志着openEuler（欧拉操作系统）正式成为开源项目。通过此次捐赠，全面开放了操作系统的源代码，以促进技术创新和生态构建。此外，华为还承诺在技术创新、生态构建、商业推广、开源建设、人才发展五个方面持续投入和推动欧拉的发展。
*   **2022年4月15日**，欧拉捐赠之后首个社区共建版本openEuler 22.03 LTS版本正式发布，该版本针对服务器、云计算、边缘计算和嵌入式四大场景首次发布新特性，方便开发者构建面向全场景的数字基础设施操作系统。
*   **2022年4月15日**，在 openEuler Developer Day 2022 上，SUSE 首次对外宣布了基于 openEuler 22.03 LTS的商业发行版——SUSE Euler Linux 2.0（2022年6月正式发布）。
*   **2023年3月30日**，发布openEuler 23.03版本，该版本基于Linux内核6.1，聚焦于硬件适应及其他技术的创新。
*   **2023年6月30日**，发布openEuler 22.03 LTS SP2，该版本提高了特定场景的算力并将系统性能提升到更高水平。
*   **2023年9月30日**，发布openEuler 22.03 LTS SP3，此版本进一步优化了系统稳定性和性能，增强了对不同计算场景的支持，提升了用户体验和操作效率。
*   **2023年9月30日**，发布openEuler 23.09创新版，采用Linux内核6.4，引入多项新功能，旨在提升开发者和用户体验。
*   **2024年6月30日**，发布openEuler 22.03 LTS SP4，此版本在保持高度稳定性的基础上，进一步增强了系统性能，特别优化了云原生和边缘计算场景下的表现。同时，新增了多项用户友好特性，显著提升了用户体验和日常操作效率。
*   **2024年**，正式发布首个AI原生开源操作系统openEuler 24.03 LTS，这一版本标志着操作系统集成AI技术的重大进化。


## openEuler 操作系统架构

openEuler 操作系统采用了与主流 Linux 发行版一致的分层架构。这种架构设计的核心思想是**隔离**与**抽象**，即将复杂的硬件操作封装起来，为上层应用提供简洁、统一、安全的接口。我们可以从两个不同的视角来理解这个架构：**程序员视角**和**最终用户视角**。

---

### 1. 程序员视角（纵向分层结构）

从开发和系统实现的角度看，系统被严格划分为**用户空间 (User Space)** 和**内核空间 (Kernel Space)**。这是现代操作系统的基石，其主要目的是实现系统保护。

<img src="https://free.picui.cn/free/2025/10/14/68ee69c9bad76.png" alt="三层架构图" title="三层架构图" />

#### 核心概念：用户空间 vs 内核空间

*   **内核空间 (Kernel Space)**：
    *   **权限最高**：运行在处理器的最高特权级别（Ring 0）。
    *   **核心职责**：直接访问并管理所有硬件资源，如 CPU 调度、内存管理、设备驱动等。
    *   **稳定性要求**：内核代码的任何错误都可能导致整个系统崩溃（Kernel Panic），因此必须稳定可靠。
    *   **主要组成**：
        1.  **系统调用接口 (System Call Interface)**：这是内核空间暴露给用户空间的**唯一、受控的**入口。用户程序不能随意进入内核，必须通过这个“大门”发起请求。
        2.  **内核核心功能 (Kernel Function)**：包括进程管理、内存管理、文件系统、网络协议栈等核心逻辑。
        3.  **体系结构相关代码**：这部分代码用于适配不同的CPU架构（如x86, ARM），实现硬件的抽象。

*   **用户空间 (User Space)**：
    *   **权限受限**：运行在处理器的较低特权级别（Ring 3）。
    *   **核心职责**：运行各种应用程序，不能直接访问硬件。
    *   **稳定性影响**：单个应用程序崩溃通常不会影响操作系统内核和其他程序的运行。
    *   **主要组成**：
        1.  **用户应用 (User Application)**：用户编写或使用的各种程序。
        2.  **GNU C 库 (glibc)**：它扮演着“中间人”的角色。大多数应用程序并不直接使用底层的系统调用，而是调用 `glibc` 提供的标准函数（如 `printf`, `fopen`）。`glibc` 内部再将这些函数调用转换为具体的**系统调用**（如 `write`, `open`），从而请求内核服务。

#### 关键交互流程：系统调用

应用程序需要读文件、创建进程或进行网络通信时，必须请求内核的帮助。这个过程如下：
1.  **应用层调用**：用户程序调用一个 `glibc` 库函数，例如 `fopen()`。
2.  **库函数封装**：`glibc` 的 `fopen()` 函数会进行一些用户空间的准备工作（如分配缓冲区），然后调用底层的 `open()` **系统调用**。
3.  **模式切换**：执行系统调用时，CPU会从**用户态**切换到**内核态**，并将控制权交给内核中对应的系统调用处理程序。
4.  **内核执行**：内核执行具体的操作（如在文件系统中查找文件、检查权限等）。
5.  **返回结果**：操作完成后，CPU从**内核态**切换回**用户态**，并将结果返回给 `glibc`，最终返回给应用程序。

#### 进程虚拟地址空间

<img src="https://free.picui.cn/free/2025/10/14/68ee69cf91157.png" alt="进程虚拟地址空间" title="进程虚拟地址空间" />

为了实现进程间的隔离，Linux 为每个进程都提供了一个独立的、私有的**虚拟地址空间**。
*   **进程隔离**：每个进程都认为自己独占了整个内存。一个进程的内存操作无法直接影响到另一个进程，保证了系统的安全性。
*   **共享内核**：如图1-2所示，所有进程的用户空间地址是独立的，但它们共享同一个**内核虚拟地址空间**。当任何一个进程通过系统调用进入内核时，它访问的都是这个公共的内核空间。

### 2. 最终用户视角（同心圆结构）

<img src="https://free.picui.cn/free/2025/10/14/68ee69d57d149.png" alt="最终用户视角" title="最终用户视角" />

从普通使用者的角度看，操作系统的结构可以理解为一个由内到外的同心圆模型，每一层都建立在内层的基础上。



*   **硬件 (Hardware)**：最核心层，是所有软件运行的物理基础，包括 CPU、内存、硬盘和各种I/O设备。
*   **内核 (Kernel)**：紧邻硬件的第二层，它是操作系统的“大脑”。它的主要工作是**管理和抽象硬件**，将复杂的硬件细节隐藏起来，为上层提供统一的接口。
*   **命令解释程序 (Shell)**：包围在内核之外的第三层。Shell 是用户与内核交互的桥梁。用户输入命令（如 `ls`, `cd`），Shell 解释这些命令，然后通过系统调用请求内核去执行相应的任务。
*   **应用层 (Application)**：最外层，是用户直接交互的各种工具和程序，如 `vi` 编辑器、`gcc` 编译器、Web服务器等。这些应用程序通过 Shell 或直接通过系统调用与内核交互。

### 归纳总结与ICT大赛备赛指南

这部分内容是操作系统部分的重中之重，理解了它，就理解了 Linux 的灵魂。

#### 1. 核心考点与必背概念

*   **用户态/内核态**：为什么要有这两种状态？（**为了保护系统**）。状态切换的开销是什么？（**上下文切换，有性能损耗**）。
*   **系统调用**：它是用户空间访问内核服务的**唯一合法途径**。请记住几个常见的系统调用，如 `fork()` (创建进程), `exec()` (执行程序), `open()` (打开文件), `read()`/`write()` (读写文件)。
*   **库函数 vs 系统调用**：这是一个高频考点。
    *   **区别**：库函数（如 `printf`）运行在用户空间，是语言或标准库层面的封装；系统调用（如 `write`）是操作系统内核提供的接口，会引起内核态切换。
    *   **关系**：一个库函数内部可能调用一个或多个系统调用，也可能不调用（如 `strcpy`）。
*   **虚拟内存**：它的作用是什么？（**进程隔离、提供比物理内存更大的寻址空间、简化内存管理**）。
*   **Shell 的角色**：它不是内核的一部分，它只是一个运行在用户空间的**应用程序**，其特殊之处在于它的主要工作是**解释用户命令并调用内核服务**。
*   **openEuler 的架构本质**：openEuler 的架构就是标准的 **Linux 宏内核 (Monolithic Kernel) 架构**。这意味着内核的大部分功能（文件系统、驱动程序、网络协议栈）都在一个单一的、巨大的内核空间中运行。
    *   **优点**：组件间通信效率高，性能好。
    *   **缺点**：一个驱动程序的 bug 就可能导致整个系统崩溃，稳定性相对差一些。

#### 2. 潜在的赛题方向

*   **选择/判断题**：
    *   “应用程序可以直接调用内核函数。” (×，必须通过系统调用)
    *   “`printf` 是一个系统调用。” (×，是C库函数)
    *   “Shell 是操作系统内核的一部分。” (×，是用户态应用程序)
    *   “Linux 采用微内核架构。” (×，是宏内核)
*   **简答/分析题**：
    *   “请简述当你在 Shell 中输入 `ls -l` 命令后，操作系统内部发生了什么？”
        *   **答题思路**：Shell (父进程) -> `fork()` 创建子进程 -> 子进程调用 `exec()` 执行 `/bin/ls` 程序 -> `ls` 程序调用 `readdir()`, `stat()` 等系统调用 -> 内核执行文件系统操作 -> 结果返回给 `ls` -> `ls` 调用 `write()` 系统调用将结果输出到屏幕。
    *   “为什么操作系统要划分用户空间和内核空间？这样做有什么优缺点？”
        *   **答题思路**：优点是**保护和安全**（隔离风险）、**稳定**（应用崩溃不影响内核）。缺点是**性能开销**（用户态和内核态之间的切换需要时间）。

#### 3. 实践启示

*   **性能优化**：理解系统调用的开销有助于编写高性能程序。例如，频繁地单字节读写文件（多次 `read`/`write` 系统调用）会比使用缓冲区一次性读写大块数据效率低得多。
*   **问题排查**：当一个程序出问题时，可以根据其行为判断问题可能出在哪个层面。是应用程序逻辑错误（用户空间），还是系统资源不足（内核管理），或是驱动不兼容（内核空间）？

掌握以上知识点，你不仅能够清晰地回答关于 openEuler 甚至所有 Linux 系统架构的问题，还能展现出对操作系统底层原理的深刻理解，这在ICT大赛中将是一个重要的加分项。
